How to Use


    

      
1

      

        
Choose your mode

        
The app has two UI modes (toggled from the topbar pill) and two device modes (selected in the input panel).

        
Walkthrough mode — guided step-by-step flow. Enter campaign IDs → SQL is generated → paste or upload results → review URL list → run. CSV downloads automatically when complete.

        
Workbench mode — direct access to the full input panel. Use the SQL ⓘ button to get the query, run it in r_ds_multiDb, paste or upload the result, configure settings, and run. Switching mid-session preserves results in memory.

        
Device mode — controls what the crawler validates. Desktop (default) crawls with a standard Mac/Chrome user agent and validates UTT, Shopify Plugin, and identity graph. iOS / Android / Both crawl with mobile user agents to trace MMP handoffs and app-store redirects. Each mode uses a different SQL query.

      

    


    

      
2

      

        
Generate your input

        
Click the SQL ⓘ button (Workbench) or follow Walkthrough step 2. The query shown automatically reflects your current device mode — switch device mode first, then open SQL to get the correct query.

        
Desktop query — returns: URL, clickid_param, campaign_id, campaign_name. Run in r_ds_multiDb.

        
Mobile query — returns: URL, clickid_param, android_clickid_param, ios_clickid_param, campaign_id, campaign_name. The mobile-specific columns extract click ID parameter names from the mobile_app_landingpages field. Run in r_ds_multiDb.

        
In mobile mode, the input must include android_clickid_param and ios_clickid_param columns — the tool validates this on paste and will show an error if the desktop query was used by mistake.

        
The tool accepts both plain CSV and MySQL table output (+---+ borders) — paste either format directly, or upload a CSV file.

      

    


    

      
3

      

        
Configure and run

        
Adjust settings if needed, select your device(s) in mobile mode, then click Run. The terminal streams live progress. Results appear as each URL completes.

        
Desktop: Identity enrichment runs automatically after the crawl — up to 15 lookups in parallel, retrying every 30s for ~5 minutes. Rows update in place. Click Stop Identity to halt early; resolved rows are preserved, unresolved rows marked N/A.

        
Mobile: iOS and Android crawl in parallel per URL. No identity enrichment — mobile results focus on redirect chain, MMP handoff, and click ID propagation.

        
All crawls use test partner ID 2222. Programs with Traffic Guard may reject this and strip the click ID. If this occurs, test with partner ID 1111 or a live link instead.

      

    


    

      
4

      

        
Read results — Desktop

        
Use the filter tabs to view by integration type or status. Click any row to open the detail drawer — showing the full redirect chain, cookies, UTT/Shopify payloads, identity graph data, and a Recommended Action when issues are detected.

        
Check results show as ✓ (pass) / ✗ (fail) / — (not applicable for this integration type).

        
When a click ID is detected, an ↗ ER button links to the internal Event Repository for that click ID. These are test clicks in an isolated incognito session — only the click record itself will be present, not impression or conversion events.

        
A ⚠ on CustomerId means a customer ID was passed in an anonymous session — this should only fire post-authentication. Flag to the client and advise using CustomProfileId instead.

      

    


    

      
5

      

        
Read results — Mobile

        
Mobile results appear under the Mobile filter tab. Each row shows per-device status, MMP, destination, and click ID for both iOS and Android. Click a row to open the mobile drawer.

        
Status badges: PASS — click ID detected and handoff/landing confirmed. WARN — handoff detected but no click ID (configuration issue). FAIL — no handoff, no MMP, no landing. SKIP — tracking link never redirected beyond the impact domain.

        
Destinations: App Store iOS / App Store Android — direct app-store redirect. MMP Link — crawler reached the MMP handoff but cannot follow through (tool limitation — manual device testing required). MMP Interstitial — Button network interstitial detected. Mobile Web — landed on a mobile web page. No Redirect — never left the impact domain.

        
Use the ▾ chevron next to the Mobile tab to sub-filter by MMP (programs with app-store or MMP handoff) or Non-MMP (mobile web and no-redirect programs).

        
The mobile drawer shows Recommended Action, Crawl Notes, General details, MMP Handoff Payload, and Redirect Chain — each section has its own Copy button. The ↗ ER button links directly to the Event Repository for the detected click ID.

      

    


    

      
6

      

        
Export results

        
In Walkthrough mode, a CSV downloads automatically when the run completes. In Workbench mode, use ↓ CSV or ⧉ Sheets — both respect the active filter tab, and both work for desktop and mobile results.

        
Sheets copies tab-separated values to the clipboard — paste directly into Google Sheets. CSV downloads a file.

        
The Copy All button in the detail drawer generates a plain-text dump of all fields for a single result — useful for Slack or tickets. The mobile drawer also has per-section copy buttons (Crawl Notes, Redirect Chain, MMP Handoff Payload).

        
Use ↻ Re-run Issues to re-crawl only failed or skipped desktop results using current settings. Results replace the original rows in place.

About Tracking Validator


    
An internal TSE tool that automates the evaluation of impact.com tracking integrations at scale. It crawls tracking links using a real headless Chromium browser — the same engine that powers Google Chrome — follows the full redirect chain, and validates the tracking implementation against a comprehensive set of checks.


    
Built to replace manual link-by-link inspection. A batch of hundreds of programs that would take hours to check manually runs in minutes, surfaces integration issues, and generates actionable remediation guidance without leaving the app.


    
How it works

    
    
      
  • Desktop crawl — each tracking link runs in a dedicated headless Chromium context (isolated, incognito, no prior state). Follows every redirect hop, intercepts network requests, executes JavaScript, captures cookies and page state after a configurable wait, then runs a full suite of checks. Results include integration type, verbatim UTT/Shopify payload from the network, click ID propagation, cookies, and redirect chain.
    • 
      
  • Mobile crawl — same Chromium engine with iOS Safari and Android Chrome user agents. Traces how the tracking link routes mobile users — detecting MMP handoffs (AppsFlyer, Branch, Adjust, Kochava, Singular, Tune, Button), app-store redirects, mobile web landings, and no-redirect cases. iOS and Android crawl in parallel per URL.
    • 
      
  • Identity enrichment — runs asynchronously after a desktop crawl. Queries impact.com's identity service using the IR_PI cookie (PRO lookup), falling back to CLI if no PRO node is found. Up to 15 lookups run in parallel, retrying every 30s for ~5 minutes. Use Stop Identity to halt early — resolved rows are preserved, unresolved rows marked N/A.
    • 
      
  • Remediation engine — a priority-ordered rule engine evaluates each completed result and generates a specific, actionable recommendation. Rules cover UTT placement and load order, Shopify plugin configuration, click ID propagation failures, Traffic Guard interference, identity graph issues, and mobile redirect configuration.
    • 
    


    
What it checks — Desktop

    
    
      
  • Click ID — present in final URL as a standalone parameter; embedded-only detection; ↗ ER button links to the Event Repository
    • 
      
  • Universal Tracking Tag — library detection and load timing; Identify call path and status; CustomProfileId, CustomerId, click ID in payload; implementation method (GTM, Tealium, Segment, Adobe Launch, direct); verbatim identify request payload
    • 
      
  • Shopify Plugin — PageLoad API request and status; click ID, CustomProfileId, and CustomerId in payload; web pixel console signal; consent API; verbatim pageload request payload
    • 
      
  • Cookies — IR_PI, IR_{campaign_id}, IR_gbd, click ID cookies
    • 
      
  • Redirect chain — full hop-by-hop with server IPs; profile redirect (ojrq.net); Traffic Guard; child/parent program routing
    • 
      
  • Identity graph — consumer ID, PRO node, FPC node, CLI node — confirming stitching end-to-end
    • 
    


    
What it checks — Mobile

    
    
      
  • Per-device status — PASS / WARN / FAIL / SKIP, independently for iOS and Android
    • 
      
  • MMP handoff — AppsFlyer, Branch, Adjust, Kochava, Singular, Tune, Button
    • 
      
  • Terminal type — App Store iOS/Android, MMP Link, MMP Interstitial, Mobile Web, No Redirect, Custom Scheme
    • 
      
  • Click ID propagation — extracted from the redirect chain using device-specific, desktop, and common fallback params
    • 
      
  • MMP handoff payload — verbatim URL params passed to the MMP at handoff
    • 
      
  • Full redirect chain — hop-by-hop with hop type classification (impact, mmp, app-store, normal)
    •

Redirect Chain Signals

    

      
SignalWhat it means

      
        
Profile Redirectojrq.net appeared — impact's browser recognition server was contacted to check for an existing brwsr cookie. In isolated test sessions, the cookie will not be dropped.

        
Traffic Guardtrafficguard.ai appeared. If click ID is also missing, test partner ID (2222) was likely rejected — test with 1111 or a live link.

        
Child/ParentA second impact tracking URL with a different campaign ID appeared — child program routing through a parent. Noted for awareness only.

        
CID EmbeddedClick ID found embedded inside another parameter's value but not as a standalone parameter. Configure a dedicated {clickid} parameter.

      
    


    
Limitations

    
    
      
  • MMP deep-links — the crawler cannot follow app deep-links. Results are marked MMP Link — a real device is required to verify the final destination.
    • 
      
  • Third-party cookies — crawls run in a fresh incognito session; the brwsr cookie will never be present. Use the Profile Redirect signal to confirm the brwsr lookup was attempted.
    • 
      
  • CAPTCHA — sites with Cloudflare or aggressive bot detection block the desktop crawler. Results are marked SKIP. Mobile crawls are generally less affected.
    • 
      
  • Server-side integrations — CAPI and trackConversion are not observable on the landing page. These appear as UNKNOWN or ClickId Integration — verify via Loggly or SQL.
    • 
      
  • Landing page only — validates what fires on the initial page load. Does not validate conversion events, postback calls, or server-side attribution logic.
    • 
      
  • Test partner ID — all generated links use partner ID 2222. Traffic Guard may reject this for some programs — test with 1111 or a live link.
    • 
    


    
Technical

    
    
      
  • Built on Electron (v30), Playwright (Chromium), Node.js
    • 
      
  • Packaged as a native macOS app (arm64 and x64 DMG)
    • 
      
  • No data leaves your machine — all crawling happens locally using the bundled Chromium browser
    • 
      
  • Identity enrichment queries impact.com's identity service directly
