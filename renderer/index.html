How to Use


    

      
1

      

        
Choose your mode

        
The app has two modes, toggled from the topbar pill. Walkthrough (default) guides you through each step in sequence. Workbench gives you the full input panel directly — for power users who already know the flow.

        
Walkthrough mode — enter campaign IDs, copy the generated SQL, paste or upload the results, review the URL list, then run. The app advances through each step automatically and downloads your CSV when complete.

        
Workbench mode — paste or upload query results directly, configure settings, and click Run. Switching to Workbench mid-session preserves any results already in memory.

      

    


    

      
2

      

        
Generate your input

        
In Walkthrough mode, the SQL query is generated for you once you enter your campaign IDs. In Workbench mode, use the SQL ⓘ button to open the query. Run it in r_ds_multiDb and paste or upload the result.

        
The tool accepts both plain CSV and MySQL table output (with +---+ borders) — paste either directly. You can also upload a CSV file.

        
CSV column order: URL, clickid_param, campaign_id, campaign_name — campaign_name is optional; if absent it will not appear in results.

        
Example: https://brand.sjv.io/c/2222/123/456,irclickid,456,Brand Name

      

    


    

      
3

      

        
Configure and run

        
Adjust settings if needed, then click Run. The terminal streams live progress. Results appear in the table as each URL completes — you don't need to wait for the full batch.

        
Identity enrichment runs automatically in the background after the crawl completes. Up to 15 lookups run in parallel, retrying every 30 seconds for up to ~5 minutes. Rows update in place as data resolves.

        
If identity enrichment is taking too long, click Stop Identity in the running banner. Results that have already resolved are preserved; unresolved rows are marked N/A. In Walkthrough mode this triggers an immediate CSV download with a _partial filename.

        
All crawls use test partner ID 2222 in the tracking URL. Some programs use Traffic Guard, which will strip the click ID when the partner ID is not recognised. If this occurs, test with partner ID 1111 or a live link instead.

      

    


    

      
4

      

        
Read results

        
Use the colour-coded filter tabs to view by integration type or status. Click any row to open the detail drawer — showing cookies, redirect chain, identity graph data, and a Recommended Action when issues are detected.

        
Check results are shown as ✓ (pass) / ✗ (fail) / — (not applicable for this integration type).

        
When a click ID is detected, an ↗ ER button appears in the results table and the detail drawer. Click it to open the internal Event Repository for that click ID — shows the click's data and metadata. Note: these are test clicks generated in an isolated incognito session; only the click record itself will be present, not impression or conversion events.

        
A ⚠ on CustomerId means a CustomerId was passed in an anonymous session — this should only fire post-authentication. Flag to client and advise using CustomProfileId instead.

      

    


    

      
5

      

        
Export results

        
In Walkthrough mode, a CSV is automatically downloaded when both the crawl and identity enrichment complete. The filename includes _complete (or _partial if identity was stopped early). In Workbench mode, use ↓ CSV or ⧉ Sheets to export manually — both respect the active filter tab.

        
The Copy All button in the detail drawer generates a plain-text dump of all fields for a single result — useful for pasting into tickets or Slack.

        
Use ↻ Re-run Issues to re-crawl only failed or skipped URLs using the current config. Results replace the original rows in place.

About Tracking Validator


    
An internal TSE tool that automates the evaluation of impact.com tracking integrations at scale. It navigates to each tracking link using a headless Chromium browser, follows the full redirect chain, and validates the tracking implementation against a comprehensive set of checks.


    
Built to replace manual link-by-link inspection — a single run can validate hundreds of programs in minutes, surface integration issues, and generate actionable remediation guidance without leaving the app.


    
How it works

    
    
      
  • Tag detection & redirect chain analysis — navigates each tracking link in an isolated Chromium session, follows every redirect hop, and captures the full chain including server IPs. Detects UTT library load, Identify call firing, Shopify PageLoad requests, web pixel console signals, Traffic Guard, profile redirect (ojrq.net), and child/parent program routing.
    • 
      
  • Cookie extraction — captures IR_PI, IR_{campaign_id}, IR_gbd, click ID cookies, and brwsr after page load. The incognito session means brwsr will not be present — the Profile Redirect signal confirms the ojrq.net lookup was attempted.
    • 
      
  • Identity enrichment — after the crawl completes, queries impact.com's identity service using the IR_PI cookie (PRO lookup). Falls back to CLI (CustomProfileId) lookup if no PRO node is found — relevant for Shopify Plugin integrations. Up to 15 lookups run in parallel, retrying every 30 seconds for a maximum of ~5 minutes. Use the Stop Identity button if enrichment is taking too long — resolved rows are preserved, unresolved rows are marked N/A.
    • 
      
  • Remediation engine — a priority-ordered rule engine evaluates each completed result and generates a specific, actionable recommendation. Rules cover integration type detection failures, UTT placement issues, Shopify plugin misconfiguration, click ID propagation failures, Traffic Guard interference, and identity graph issues.
    • 
    


    
What it checks

    
    
      
  • Click ID — present in final URL as a standalone parameter; whether it appears only embedded inside another parameter's value; click IDs link to the internal Event Repository showing the click's data and metadata
    • 
      
  • Universal Tracking Tag — JavaScript library detection and load timing; the UTT library should be placed in <head> and the Identify call high in <body> after the library has loaded; validates Identify call path, status, CustomProfileId, CustomerId, and click ID in payload
    • 
      
  • Shopify Plugin — PageLoad API request, web pixel console signal, consent API, integration source, click ID and CustomProfileId in payload
    • 
      
  • Cookies — IR_PI, IR_{campaign_id}, IR_gbd, click ID cookies, brwsr
    • 
      
  • Implementation method — GTM, Tealium, Segment, Adobe Launch, or direct
    • 
      
  • Redirect chain — full hop-by-hop with server IP addresses, profile redirect (ojrq.net), Traffic Guard, child/parent program routing
    • 
      
  • Identity graph — enrichment via identity service API; consumer ID, PRO node (click's profile ID), FPC node (first-party cookie), CLI node (CustomProfileId cookie) — confirming identity stitching end-to-end
    • 
      
  • Remediation — rule-based actionable recommendations generated automatically for each result
    •

Redirect Chain Signals

    

      
SignalWhat it means

      
        
Profile Redirectojrq.net appeared in the redirect chain — impact's browser recognition server was contacted. When no brwsr cookie is present, impact redirects to ojrq.net to check for an existing cookie; if none is found, a fresh cookie is generated and set on the tracking domain. In isolated test sessions the cookie will not be dropped.

        
Traffic Guardtrafficguard.ai appeared in the redirect chain. If click ID is also missing from the final URL, the test partner ID (2222) was likely rejected — test with 1111 or a live link.

        
Child/ParentA second impact tracking URL with a different campaign ID was found in the redirect chain, indicating a child program routing through a parent program. Noted for awareness — no action required.

        
CID EmbeddedThe click ID value was found embedded within another URL parameter (e.g. sourceid=imp_{clickid}) but not as a standalone parameter. The program should be configured with a dedicated {clickid} parameter.

      
    


    
Limitations

    
    
      
  • Crawls run in a fresh incognito session — brwsr is set by the tracking domain as a third-party cookie, but the incognito session prevents it from being dropped
    • 
      
  • CAPTCHA-protected sites will block the headless browser — manual validation required for those results
    • 
      
  • All links use test partner ID 2222 — Traffic Guard may reject this for some programs; test with 1111 or a live link if this occurs
    • 
      
  • Sites with aggressive bot detection may time out or return unexpected results even when the integration is working correctly
    • 
      
  • Custom or server-side integrations (CAPI, trackConversion) are not detectable on the landing page and will appear as UNKNOWN or ClickId Integration
    • 
      
  • Stopping identity enrichment early marks unresolved rows as N/A — re-running the crawl will trigger a fresh enrichment attempt
    •
